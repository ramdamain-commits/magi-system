# MAGI System Update Policy

## 1. このアプリの目的

MAGI System は、AI に判断を委任するアプリではない。異なる価値観をぶつけて、盲点、反対理由、実行条件を露出させるための意思決定支援 UI である。

目的は次の 3 点に集約する。

- 対立を見せる
- 条件付きの意思決定を見せる
- 人間が最終判断しやすい材料を整える

## 2. 何を優先するか

今後の更新では、次の優先順位を守る。

1. 審議が割れること
2. 割れた理由が読めること
3. 合議結果から次の行動が分かること
4. そのうえで繰り返し遊べること
5. そのうえで見た目が強いこと

見栄えだけが増えて、審議の差分が弱くなる変更は避ける。

## 3. パネル設計の原則

### BALTHASAR

- 役割: 前提破壊
- 主に問うもの: 常識依存、楽観、言い換えだけの改善、権威への依存
- 承認しやすい条件: 挑戦する価値が高く、失敗しても引き返せる

### CASPAR

- 役割: 被害監査
- 主に問うもの: 権利侵害、悪用、炎上、弱者へのしわ寄せ、説明責任
- 承認しやすい条件: 保護策、監視体制、責任の所在が具体的

### MELCHIOR

- 役割: 執行責任
- 主に問うもの: 実装可能性、責任者、コスト、運用負荷、撤退計画
- 承認しやすい条件: 小さく試せて、失敗時に止められる

### 禁止事項

- 3 パネルを似た性格に寄せること
- 中立や要約に逃がして衝突を弱めること
- 結論だけ一致して理由が似る状態を放置すること

## 4. 出力フォーマット契約

1 回のリクエストで、次の JSON 構造を返させる。

- `panels.b`
- `panels.c`
- `panels.m`

各パネルは、最低限次の要素を必ず含める。

- `conclusion`
- `reasons`
- `risk`
- `conditions`
- `verdict`

- `reasons` は最大 1 件、`conditions` は最大 1 件に抑える。

合議は API に必須で返させず、UI 側で次の情報を合成する。

- `final_text`
- `vote_summary`
- `majority_view`
- `minority_view`
- `hold_conditions`
- `recommended_action`

モデルが `consensus` を返すこと自体は許容するが、UI は `panels` だけで表示を成立させること。受信側コード（`buildValidatedDeliberation` 等）は `payload.consensus` を読まず、`panels` のみを正規入力として扱う。

この契約を変える場合は、`index.html` 内のバリデーションと README を同時に更新する。

## 5. 合議 UI の契約

合議エリアでは、最低限次を表示し続ける。

- 投票結果
- 最終判定
- 多数派の要約
- 少数意見
- 保留条件
- 推奨アクション

多数決だけを大きく見せて、少数意見や条件を隠す変更は行わない。

## 6. API 負荷軽減の原則

- 1 回の問いにつき、外部 API 呼び出しは原則 1 回にする
- 3 パネルの時間差表示は UI 演出で行い、通信を 3 本に増やさない
- 出力は短い構造化レスポンスに制限し、`理由` は最大 1 件、`承認条件` は最大 1 件に抑える
- `RESET` は進行中のリクエストを中断できるようにする
- 自動再試行は安易に増やさない。必要なら人間が再実行する
- API 負荷軽減のために人格差を弱める変更は行わない
- `DEMO` モードを置く場合は、疑似審議であることを UI 上で明確に示し、実AIの返答と誤認させない
- 実験モードは「実験的」ラベル付きで user-facing に出してよいが、品質基準未達時は DEMO へ自動フォールバックすること
- 実験モードの安定基準は、代表質問 5 件で安定成功し、内容も日本語で最低限読めること

## 7. ゲーム性を高める方針

- 良い問いの例を UI に出し、衝突しやすい入力を促す
- 満場一致より 2 対 1 の緊張感を歓迎する
- 条件付き承認、差し戻し、保留など、結果のグラデーションを維持する
- 「誰が何を嫌がっているか」を読み物として楽しめる構造にする
- 単発 API 呼び出しでも、開示順と演出で審議感を保つ

## 8. エラー案内の原則

- 入力ミスや通信失敗は `alert()` へ逃がさず、画面内のバナーや入力欄下で次の行動を示す
- `API キー未入力`、`認証失敗`、`瞬間レート超過`、`無料枠到達`、`安全フィルター` は区別して案内する
- 実験モードの失敗要因は通常導線に混ぜず、必要なら `docs/HANDOFF.md` に KPT として残す
- グローバル障害はパネルごとに重複表示せず、上部で 1 回だけ説明する

## 9. ドキュメント更新方針

機能追加や仕様変更のたびに、少なくとも次を確認する。

- `README.md`
- `CHANGELOG.md`
- `docs/UPDATE_POLICY.md`
- 必要なら `CLAUDE.md`
- GitHub PR は日本語で書き、`.github/pull_request_template.md` を使う
- テキストの文字コードは `.editorconfig` を正とし、Markdown と PowerShell は UTF-8 BOM、その他のテキストは UTF-8 を基本にする
- PR では `.github/workflows/text-encoding-check.yml` が `scripts/Test-TextEncoding.ps1 -Recurse -FailOnWarning` を実行する

更新が不要な場合も、なぜ不要かを作業報告で短く説明する。

## 10. 非目標

現時点での非目標は以下。

- 厳密な業務意思決定システムとしての保証
- モデル比較ベンチマーク
- 長文レポート生成ツール化
- フレームワーク導入やビルド工程の追加
